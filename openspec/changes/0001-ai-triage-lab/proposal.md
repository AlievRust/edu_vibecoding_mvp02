# Proposal: ai-triage-lab

## Problem

В репозитории есть только исходный промпт и минимальный каркас документации. Нет описания целевого поведения, структуры приложения, API, хранения данных и сценариев отказа. Без OpenSpec реализация учебного проекта рискует уйти в догадки и несогласованные решения.

## Goal

Сформировать полный OpenSpec для учебного FastAPI MVP `ai-triage-lab`, который:

- принимает обращения через `POST /triage`;
- классифицирует их по категориям `billing`, `support`, `complaint`, `other`;
- пишет аудит в SQLite;
- ограничивает число запросов на `client_id`;
- поддерживает mock-режим, режим отказа и интеграцию с Yandex Alice;
- запускается локально и в Docker;
- покрывается pytest-тестами и документацией.

## Scope

В scope входят:

- проектный контекст `openspec/project.md`;
- OpenSpec change с `proposal.md`, `design.md`, `tasks.md`;
- структура FastAPI-приложения;
- схемы запроса и ответа;
- SQLite-таблицы для аудита и лимитирования;
- модуль классификации `mock`;
- модуль интеграции `aliceai-llm`;
- сценарий отказа `broken`;
- Dockerfile и `docker-compose.yml`;
- `scripts/inspect_db.py`;
- pytest-тесты;
- обновление `README.md` и `.env.example`.

## Out of scope

- авторизация и аутентификация;
- веб-UI;
- очереди задач и фоновые процессы;
- внешняя БД кроме SQLite;
- миграционный фреймворк;
- production-grade observability;
- сложная оркестрация нескольких моделей;
- ручная модерация через отдельный интерфейс.

## User-visible behavior

- `POST /triage` принимает JSON с `client_id`, `channel`, `text`.
- Пустой `text` и слишком длинный `text` отклоняются.
- Успешный запрос возвращает категорию, confidence, `escalate`, `draft_reply` и `error`.
- При недоступности LLM или невалидном ответе возвращается fallback-результат: `category=other`, `confidence=low`, `escalate=true`, безопасный текст ответа, `error` заполнен.
- При превышении лимита возвращается HTTP 429.
- Каждое обращение сохраняется в SQLite, включая fallback-сценарии и rate-limit-отказы.
- Пользователь может просмотреть последние записи через `scripts/inspect_db.py`.

## Risks

- SQLite может стать узким местом при конкуренции за запись.
- Интеграция с Yandex Alice зависит от внешнего API и ключей окружения.
- Требование сохранять аудит для всех веток обработки усложняет обработку ошибок и валидации.
- Нужен явный выбор максимальной длины `text`, потому что исходный промпт её не задаёт.
- Неправильно оформленный JSON от LLM может потребовать строгой нормализации ответа.

## Acceptance criteria

- OpenSpec описывает целевой MVP без неявных допущений.
- Реализация будет иметь `POST /triage`, SQLite-аудит, rate limit, mock/broken/Alice provider, Docker, тесты и README.
- Сценарии успеха, отказа и превышения лимита зафиксированы в спеках.
- Все изменения можно проверить локально с помощью документированных команд.
